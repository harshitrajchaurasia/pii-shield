## 1. Initializing a Git Repository

PII Shield started as a local Python project in `D:\dev\projects\PI_Removal` — no git, no version control. Before pushing to GitHub, we needed to:

1. Create `.gitignore` (BEFORE `git init`, so nothing bad gets tracked)
2. Clean junk files (caches, logs, test data)
3. Remove hardcoded secrets from source code
4. Then `git init` and make the first commit

**Why this order matters:** Once a file is committed to git, it lives in git history forever (even if you delete it later). If you commit a secret, you'd need to rewrite git history (`git filter-branch` or BFG Repo Cleaner) to truly remove it. Much easier to clean first.

```bash
# Step 1: Create .gitignore first
# Step 2: Clean junk files
# Step 3: Replace secrets with placeholders
# Step 4: Then init
git init
git add .
git commit -m "Initial commit: PII Shield - LLM Safety Middleware"
```

## 2. Designing `.gitignore`

Here's our actual `.gitignore`, section by section:

### Python Artifacts
```gitignore
__pycache__/          # Compiled bytecode (auto-generated, OS-specific)
*.py[cod]             # .pyc, .pyo, .pyd files
*.egg-info/           # Package metadata (generated by pip install -e)
dist/                 # Built distributions
build/                # Build artifacts
*.egg                 # Egg packages
```
**Why:** These are generated files. Every developer's machine creates different ones. Never commit generated files.

### Tool Caches
```gitignore
.mypy_cache/          # Type checker cache
.pytest_cache/        # Test runner cache
.ruff_cache/          # Linter cache
htmlcov/              # Coverage HTML reports
```

### Logs
```gitignore
*.log                 # Any log file
logs/                 # Log directory
```
**Why for a PII project:** Even though we have PI-safe logging (a log filter that redacts PII from log output), belt-and-suspenders approach — never commit logs.

### Secrets and Credentials — THE MOST IMPORTANT SECTION
```gitignore
.env                  # Environment variables (contains real secrets)
.env.*                # All .env variants (.env.dev, .env.prod, etc.)
!.env.example         # EXCEPTION: commit the template (shows what vars are needed)
!.env.*.template      # EXCEPTION: commit templates too
CREDENTIALS.txt       # A file that literally had JWT secrets in it
config/clients.yaml   # API client credentials (real IDs and secrets)
```

**The `!` negation pattern:** `.env.*` excludes ALL files starting with `.env.` — but `!.env.example` says "except this one, DO track it." This lets you commit a template file showing developers what environment variables they need, without committing the actual values:

```bash
# .env.example (committed — safe, shows structure)
JWT_SECRET=REPLACE_WITH_GENERATED_SECRET
CLIENT_ID=your-client-id
CLIENT_SECRET=your-client-secret

# .env (NOT committed — has real values)
JWT_SECRET=zn_OCwQX9KpT9WBok6GlBJ1GpIhdSsUfGGa69h-X5T4
CLIENT_ID=pi-web-service
CLIENT_SECRET=_wRlqZ_YQ1QVuIPrMl5c5sDdXNRyHS7nOmpf3S6SEEo
```

### Data Files That May Contain PII
```gitignore
*.csv                 # CSV files (we had 556MB of real support ticket data!)
*.xlsx                # Excel files
*.xls                 # Old Excel format
data/*.json           # JSON data files
!data/names.json      # EXCEPTION: the name dictionary (safe, no real PII)
```

**Why `!data/names.json`:** The redaction engine needs a dictionary of common names (Indian names, company names) to detect them in text. This file contains generic names like "Amit", "Priya", "Kumar" — not real people's data. It's safe and required for the app to work.

### Legacy and Misc
```gitignore
others/               # Old backups, standalone scripts, test data, misc files
*.postman_collection.json  # API testing collections (may contain real request data)
```

## 3. Secret Scanning and Cleanup

### The Problem

We found **5 distinct secret values** hardcoded across **60+ files** — in Python source code, Docker Compose files, shell scripts, documentation, and test files:

| Secret | Type | Where Found |
|--------|------|-------------|
| `zn_OCwQX9KpT...` | JWT signing secret (dev) | Config files, scripts, docs |
| `-iFfgAhYWT82...` | JWT signing secret (prod) | Config files, scripts, docs |
| `_wRlqZ_YQ1QV...` | Client credential (dev) | Docker Compose, config, scripts |
| `8rwexUor9eUp...` | Client credential (prod) | Docker Compose, config, scripts |
| `VjK8mNpL2qR9...` | Web client secret | Dockerfiles, config |

### The Fix: Bulk Find and Replace

```bash
# Replace each secret with a descriptive placeholder
find . -path ./others -prune -o -type f -print | \
  xargs sed -i 's/zn_OCwQX9KpT9WBok6GlBJ1GpIhdSsUfGGa69h-X5T4/YOUR_DEV_JWT_SECRET_HERE/g'

find . -path ./others -prune -o -type f -print | \
  xargs sed -i 's/-iFfgAhYWT82x9mpQKotjzBMJ6E8VJCz2ChMF15BJF4/YOUR_PROD_JWT_SECRET_HERE/g'

# ... repeated for all 5 secrets
```

**Key decisions:**
- `-path ./others -prune -o` — skips the `others/` directory (being deleted anyway)
- Placeholder names like `YOUR_DEV_JWT_SECRET_HERE` are self-documenting
- After replacement, `config/clients.yaml` (with real creds) is gitignored; committed files only have placeholders

### What We Deleted
- `others/CREDENTIALS.txt` — contained JWT secrets in plain text
- `docker/.env.dev`, `docker/.env.prod` — real environment files with secrets
- `others/standalone/*.csv` — 556MB of real support ticket data containing actual PII
- All `__pycache__/` directories (6 total)
- All `*.log` files (5 total)

## 4. GitHub CLI (`gh`)

### Creating the Repository

```bash
# Create a public repo on GitHub
gh repo create ErosiousHarsh/pii-shield --public --source=. --remote=origin

# Or if you prefer two steps:
gh repo create ErosiousHarsh/pii-shield --public
git remote add origin https://github.com/ErosiousHarsh/pii-shield.git
```

### Pushing Code

```bash
git push -u origin master    # -u sets upstream tracking
```

### Managing Secrets (used for CI/CD)

```bash
# Set a secret (value is not shown in UI after creation)
gh secret set WIF_PROVIDER --body "projects/49982461185/locations/global/..."
gh secret set WIF_SERVICE_ACCOUNT --body "github-deployer@shared-project-489320.iam.gserviceaccount.com"
gh secret set GCP_PROJECT_ID --body "shared-project-489320"

# List secrets (shows names and creation dates, NOT values)
gh secret list
# WIF_PROVIDER         2026-03-21T13:51:57Z
# WIF_SERVICE_ACCOUNT  2026-03-21T13:51:58Z
# GCP_PROJECT_ID       2026-03-21T13:52:10Z
```

### Monitoring Workflow Runs

```bash
# List recent workflow runs
gh run list --limit 5

# View details of a specific run
gh run view 23381266469

# See ONLY the failed logs (key for debugging)
gh run view 23381266469 --log-failed

# Re-run only the failed jobs (saves time, no need to re-push)
gh run rerun 23381266469 --failed
```

More on this in [Doc 04 — GitHub Actions CI/CD](04-github-actions-cicd.md).

## 5. Branch Naming: `main` vs `master`

### The Problem

When you run `git init`, git creates a default branch called `master` (unless you've configured `init.defaultBranch`). But GitHub's web UI defaults to `main` for new repos.

**This matters because CI/CD workflow files specify which branch triggers them:**

```yaml
# .github/workflows/ci.yml
on:
  push:
    branches: [master]    # <-- Must match your actual branch name!
```

If your branch is `master` but the workflow says `main`, **the workflow will never trigger**. Nothing will break visibly — you just won't see any CI/CD runs.

### What Happened

1. We ran `git init` → created `master` branch
2. We initially wrote workflows with `branches: [main]` (GitHub convention)
3. Pushed to GitHub → workflows didn't trigger
4. Checked: `gh repo view --json defaultBranchRef` → showed `master`
5. Fixed: changed all workflow files from `[main]` to `[master]`

### How to Avoid This

Either:
```bash
# Option A: Set git default globally to 'main'
git config --global init.defaultBranch main

# Option B: Rename after creation
git branch -M main
```

## 6. Troubleshooting

### `fatal: detected dubious ownership in repository`

**When:** Running any git command on Windows.

**Cause:** The directory was created by one Windows user account (SID), but git is running as a different user (e.g., elevated terminal vs normal terminal). Git refuses to operate as a safety measure against malicious repos.

**Fix:**
```bash
git config --global --add safe.directory D:/dev/projects/PI_Removal
```

### `gh: command not found` (in bash on Windows)

**When:** Running `gh` in Git Bash or WSL.

**Cause:** GitHub CLI was installed for Windows (via Chocolatey/MSI), which adds it to the Windows PATH. But Git Bash uses a different PATH and doesn't inherit the Windows PATH entry.

**Fix:** Use the full Windows path with forward slashes:
```bash
"/c/Program Files/GitHub CLI/gh.exe" repo create ...
"/c/Program Files/GitHub CLI/gh.exe" secret set ...
```

Or add it to your `.bashrc`:
```bash
echo 'alias gh="/c/Program Files/GitHub CLI/gh.exe"' >> ~/.bashrc
```

### `refreshenv` doesn't work in bash

**Context:** After installing `gh` via Chocolatey, the user tried `refreshenv` to reload PATH.

**Why it fails:** `refreshenv` is a Chocolatey PowerShell function. It doesn't exist in bash. In bash, you'd need to close and reopen the terminal, or manually export the new PATH.

## Key Takeaways

1. **Create `.gitignore` BEFORE your first commit** — removing secrets from git history is painful
2. **The `!` negation pattern** in `.gitignore` lets you commit templates (`.env.example`) while excluding real secrets (`.env`)
3. **Search for secrets systematically** before going public — they hide in docs, scripts, tests, and Docker files
4. **Branch naming must match workflow triggers** — `main` vs `master` is a silent CI/CD killer
5. **`gh` CLI is powerful** — `gh secret set`, `gh run view --log-failed`, and `gh run rerun --failed` save hours of clicking through the GitHub UI
